# RECOVERY

## Core Rule

특정 채용 사이트의 HTML 클래스 리뉴얼이나 스크래핑 오류 등의 검증 실패가 발생했을 때, 무리하게 대대적인 소스코드 전면 수정을 감행하지 마십시오. 반드시 구조적 결함 진단을 먼저 행하고 최소 범위의 수정을 점진적으로 적용하는 것을 기본 원칙으로 삼습니다.

## Failure Loop

크롤러 파이프라인 가동 혹은 검증 단계에서 실패(Fail)나 타임아웃이 일어나는 경우:

1. 터미널 및 가상 러너의 에러 로그를 끝까지 전부 읽고 예외 호출 스택(Traceback)을 확인합니다.
2. 실패의 핵심적인 요인을 정확히 범주화합니다:
   - **웹 구조 변경**: 원티드/사람인 또는 게임사 공식 채용 페이지 리뉴얼로 인해 HTML 태그 셀렉터(XPath/CSS Selector)가 바뀐 경우
   - **네트워크 / 차단**: 스크래핑을 수행하는 GitHub Actions IP 주소가 일시적으로 블로킹되었거나 차단된 경우
   - **DB 스키마 불합치**: SQLite에 테이블이 올바르게 생성되지 않았거나 컬럼 타입 제약 조건에 어긋나는 한글 데이터가 삽입되려 한 경우
   - **환경 변수 결여**: 로컬 `.env` 혹은 GitHub Secrets 내에 `TELEGRAM_BOT_TOKEN` 값이 주입되지 않아 전송 에러가 난 경우
3. 오류 영역을 `PRD.md`, `PLAN.md`, `VALIDATION.md` 내용과 직접 조목조목 비교합니다.
4. 해당 결함에 대해서만 **되돌리기 쉬운 가장 미세하고 간결한 단위의 패치**를 적용합니다.
5. 수정 사항이 발생한 모듈에 대해서만 고정 유닛 테스트(`tests/test_...py`)를 최우선적으로 가동하여 수정을 완비합니다.
6. 진행 상황과 실패 및 극복 스토리를 `PROGRESS.md`에 투명하게 한 행 기록합니다.

## Retry Limit

만약 동일한 모듈의 수집 오류나 인코딩 깨짐 등의 핵심 검증 단계가 **서로 다른 3회의 수정 시도 후에도 지속적으로 실패**하는 경우:

골이 스스로 소스코드를 뜯어고치는 자가 수정 및 무한 루프 행위를 즉시 전면 멈춥니다. `/goal pause`는 Claude Code CLI에서 내장 지원되지 않으므로, 에이전트는 즉시 전체 동작을 수동 진단 및 대기 모드로 전환하고 사람(구직자/개발자)에게 결정 및 방향 피드백을 직접 요구해야 합니다. 3회 실패 시 반드시 아래 보고 양식을 포함해 응답을 완료합니다.

**Report 할 내용:**
- 실패를 야기하는 직접적인 실행 명령 및 세부 검증 불합격 컬럼/크롤러 명칭
- 지금까지 3회 동안 각각 다르게 시도했던 코드 변경 사항 및 상세 패치 내역
- 3회의 시도가 각각 실패한 원인 분석 및 구체적 에러 콘솔 요약
- 이 시점에서 가장 안전하고 지속 가능한 폴백 방안 목록 (예: "해당 게임사 크롤러 비활성화 처리 후 수동 링크 추가 등")
- 기획서(PRD) 혹은 분류 규칙의 타협이 필요한지에 대한 최종 요구 가이드

## Scope Control

다음 행위는 철저하게 통제하며 사전에 허용되지 않은 가동 방식입니다:
- 스크랩 수집 오류 수정을 빌미로, 이번 목표와 무관한 데이터베이스 관리 도구를 별도로 연동하거나 무거운 웹 관리자 서버 구조를 자체 빌드하지 않습니다.
- 크롤러 수집 차단을 강제로 뚫겠다며 유료 유료 프록시 패키지나 검증되지 않은 무거운 셀레늄(Selenium) 의존 패키지를 마구 임포트하거나 package/requirements 버전을 뒤흔들지 마십시오.
- 채용공고의 연차 파싱이나 연봉 정보 분류가 까다롭다는 이유로 특정 필드 값을 공백 처리(Silencing)해 버리거나 가짜 모의 데이터를 넣는 꼼수를 쓰지 마십시오.
- 단위 테스트(`tests/`)가 계속 깨진다는 이유로 해당 테스트 코드를 지우거나 스킵 데코레이터(`@unittest.skip`)를 걸어서 빌드 체크를 강제 패스하지 마십시오.

## Reorientation Rule

동작 방향이나 분석 기술 스택을 전격 교체하기 전에 스스로에게 질문하십시오:
1. "게임사 재무 공고를 무비용으로 나에게 데일리 알림한다"는 원형의 초심 골 문장을 다시 자세히 정독하십시오.
2. `PRD.md`의 안 만드는 것(Non-goals) 목록을 다시 정독하고 엉뚱한 직군 스크래퍼를 만들려 하지 않는지 확인하십시오.
3. 현재 마일스톤 단계(`PLAN.md`)를 다시 조명하고, 내가 하는 편집이 지금 마일스톤의 요구 조건을 100% 만족하는 직접적인 행위인지 교정하십시오.

## Revert Rule

자신이 가동한 최근 코드가 검증 상태를 더욱 최악으로 만들었거나, 원래 잘 되던 타 사이트 크롤러까지 오동작하게 한 경우에 한해 즉각적인 깃 되돌리기(`git checkout` 또는 `git revert`)를 적용하십시오. 단, 사용자가 프로젝트 시작 전 직접 구현해 놓았던 환경 설정이나 기존 세팅부는 사람의 직접 지시 없이는 절대 역으로 되돌리거나 덮어써서는 안 됩니다.
