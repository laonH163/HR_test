왜 이렇게 나눴나 — 사이트마다 방식이 다르다

"그냥 다 똑같이 긁으면 되는 거 아닌가"가 가장 흔한 오해다. 실제로는 사이트가 정보를 내주는 방식이 제각각이라, 수집 방법도 따라서 달라진다. 무거운 방법일수록 느리고 잘 막히므로, 가능하면 가벼운 방법을 쓴다.

먼저 짚고 갈 것

세 방식 모두 "화면을 사진으로 찍어 읽는" 게 아니다

웹페이지의 HTML은 글자로 된 설계도다. 브라우저가 그 설계도를 읽어서 우리 눈에 보이는 화면을 그린다. 우리는 그려지기 전의 글자 뭉치를 그대로 받아온다. 거기엔 정보마다 이름표가 붙어 있어 ("여기부터 여기까지가 공고 제목") 원하는 부분만 정확히 꺼낼 수 있다 — 이것이 파싱이다. 화면을 이미지로 찍어 글자를 알아내는 방식(OCR)은 느리고 틀리기 쉬워 쓰지 않는다. 잡코리아 상세 내용도 처음엔 OCR이 필요해 보였지만, 숨은 프레임 안에 글자로 존재하는 것을 찾아내 해결했다.

수집이 실패해도 데이터가 안 깨지는 이유 — 4중 방어선

자동화에서 제일 위험한 사고는 "못 긁었는데 공고가 사라진 줄 알고 전부 마감 처리"하는 것이다. 그래서 "오늘 안 보임 = 마감"이라고 바로 믿지 않고, 네 겹으로 의심한다.

1
접속 실패한 사이트는 판정 제외 그 사이트가 아예 안 열렸으면, 거기 공고는 "오늘 못 본 것"이지 "없어진 것"이 아니다.
2
0건이면 판정 보류 접속은 됐는데 결과가 0건이면 검색 기능이 고장 났을 가능성이 크다. 마감 처리를 미룬다.
3
일부만 성공해도 판정 보류 검색어 4개 중 3개가 막히고 1개만 통과한 경우 — 다 훑었다고 볼 수 없으므로 역시 보류한다.
4
한꺼번에 우수수 사라지면 정지 평소보다 수집량이 급감하거나 공고가 무더기로 사라지면, 자동 처리를 멈추고 사람에게 알린다.

자동 시동

GitHub의 예약 실행 기능이 정해진 시각에 깨운다. 내 컴퓨터가 꺼져 있어도 돌아간다. 주말과 한국 공휴일은 자동으로 건너뛴다.

실패 시

다른 회선으로 재도전

차단은 보통 접속 위치(IP) 단위로 걸린다. 그래서 실패한 사이트가 있으면 새 컴퓨터·새 회선에서 전체를 한 번 더 돌린다.

종료 후

결과를 저장소에 커밋

갱신된 DB와 대시보드 파일이 자동으로 저장되고, 웹페이지가 곧바로 새 내용으로 갱신된다.

실제 사례 — "원티드만 매일 0건"을 추적한 과정

2026-07-22 · 문제 해결 절차 예시

증상만 보고 코드를 고치면 틀린다

한 사이트만 계속 0건이었다. 언뜻 "수집기가 고장 났다"로 보이지만, 순서대로 확인해 보니 코드는 멀쩡하고 접속 위치가 차단당한 것이었다. 자동화 운영에서 원인을 좁혀가는 전형적인 순서다.

  1. 언제부터인가 — 실행 이력을 뒤져 정상이던 마지막 날을 찾는다. (7/16 이후 전부 0건)
  2. 내 컴퓨터에서도 그런가 — 같은 코드를 집에서 돌려본다. → 정상 작동. 코드 문제 아님이 확정된다.
  3. 다른 통로는 열려 있나 — 같은 사이트의 다른 접근 경로도 시험한다. → 자동 실행 환경에서는 전부 막힘.
  4. 차단 사유가 뭔가 — 거절 메시지 원문을 읽는다. → 지역 차단이 아니라 자동화 차단 규칙에 걸린 것.
  5. 결론 — 코드로는 못 고친다. 대신 ① 고칠 수 없는 실패를 매일 경고로 띄우지 않게 정리하고 ② 차단이 풀리면 자동으로 알려주도록 감시를 붙였다.
