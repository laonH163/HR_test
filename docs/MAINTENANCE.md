# 유지보수 기록 — 구조 결정·함정·검증 루틴

이 문서는 미래의 유지보수자(사람이든 AI 세션이든)가 **같은 삽질을 반복하지 않도록**
핵심 결정의 "왜"와 알려진 함정을 기록한다. 코드가 진실이고, 이 문서는 지도다.

## 텔레그램 경고별 대처법 (운영자용 — 이것만 보면 됨)

일상 유지보수는 "매일 브리핑 최하단의 **🩺 수집 상태** 한 구역만 훑고, 경고가 보이면
아래 표대로"가 전부다. 점검이 필요한 경고는 이 프로젝트 폴더에서 Claude Code를 열고
표의 문장을 말하면 된다. "🩺 수집 상태: 전 소스 정상"인 날은 할 일이 없다.
(경고는 메인 콘텐츠를 해치지 않도록 브리핑 최하단에 모여 있다.)

| 🩺 수집 상태의 경고 | 의미 | 대처 |
|---|---|---|
| ⚠️ 접속 실패: … | **오늘 어느 시도에서도** 그 소스를 확보하지 못함(IP 차단·네트워크). 같은 날 앞선 시도에서 이미 수집된 소스는 이 경고에서 자동 제외된다(2026-07-16 보정 — 재시도 러너만 차단된 날의 오경보 방지) | 하루는 관망(다음 날 자동 복구가 흔함). **이틀 연속이면** → "텔레그램에 접속 실패 경고가 이틀째야, 원인 점검해줘" |
| ⚠️ 검색 0건: … | 접속은 되는데 검색 결과가 빈손 — 검색 열화/개편 의심 (원티드 한 달 무음 사례가 이 유형) | **이틀 연속이면** → "OO가 이틀째 0건이야, 사이트 바뀌었는지 점검해줘" |
| ⚠️ 공고 일괄 소멸 의심: … | 한 회사(또는 여러 소스 동시)의 기존 공고가 한 번에 전부 사라짐 — 사이트 개편 의심, 마감은 보류 중 | **이틀 연속이면** → "OO 일괄 소멸 경고가 반복돼, 진짜 마감인지 개편인지 확인해줘" (진짜 전부 마감이었다면 그렇게 보고되고 정리됨) |
| 📉 수집량 급감: … | 평소의 30% 미만으로 수집량이 떨어짐 — 서서히 죽는 신호 | 지속되면 → "사람인 수집량이 계속 급감이래, 점검해줘" |
| ✅ 차단 해제 확인: … | **알려진 차단이 풀렸다** — 그 소스가 다시 수집되기 시작함 | → "OO 차단이 풀렸대, known_blocks에서 등록 해제해줘" (등록을 남겨두면 진짜 고장이 나도 정보 줄로 묻힌다) |
| ⚠️ OO 차단 N일째 — 수동 확인 필요 | 알려진 차단이 14일 넘게 이어짐. 그 소스의 활성 공고는 마감 보류로 계속 보호되므로, 실제로는 마감됐는데 살아 있는 **좀비**일 수 있음 | → 해당 소스 활성 공고 링크를 직접 열어보고, 마감됐으면 "OO 공고 마감 처리해줘" |
| ℹ️ OO: … — 조치 불요 | 경고가 아니라 **정보**. 코드로 못 고치는 차단으로 등록돼 있고 자동 복구를 감시 중 | 없음 (복구되면 ✅로, 장기화되면 ⚠️로 자동 전환된다) |
| 🩺 수집 상태: 전 소스 정상 | 정상 | 없음 |

공통 원칙: 보류 장치들 덕에 **경고가 떠 있는 동안에도 대시보드 데이터는 손상되지 않는다.**
하루 이틀 늦게 대응해도 잃는 것은 신규 공고의 신선도뿐이다. 스크래퍼가 고쳐지면
놓친 공고는 재수집으로, 잘못 마감된 공고는 자동 부활(upsert)로 스스로 복구된다.

## 필수 검증 루틴 (수정 전 반드시 읽기)

1. **스크래퍼 변경 시**: 실제 실행 결과로 검증한다. 수집 건수가 기존 대비 급감하면
   원인 규명 전 배포 금지. (예: `python -c "from src.scraper.gamejob_scraper import ...; print(len(jobs))"`)
2. **필터(filters.py) 변경 시**: DB 전체 제목으로 BEFORE/AFTER 전수 대조 —
   새로 제외/통과되는 제목 목록을 눈으로 확인한다. 2026-07-09 개발직 차단 때
   83건 대조로 부수 피해 0을 확인하고 배포한 것이 표준 절차.
3. **중복 병합 키 변경 시**: `src/utils/dedup.py`(Python)와
   `templates/dashboard_template.html`의 `processRawJobsData()`(JS)가 **동일 로직으로
   미러링**되어 있다. 한쪽만 고치면 텔레그램과 대시보드의 병합 결과가 갈라진다.
   반드시 양쪽을 함께 수정하고, 실DB로 Python 결과와 생성된 index.html의 JS 실행
   결과(Node)를 대조한다.
4. **커밋 전**: `git diff --cached`로 민감정보(텔레그램 토큰·chat_id·개인정보) 스캔.

  패턴("[회사명] 제목 채용 공고 | 원티드")만 신뢰. 탈락 시 `[WANTED DROP]` 로그.
- CI 러너에서는 검색 결과가 로컬보다 적게 렌더링될 수 있다(IP·지역 요인 추정).

### 게임잡
- **2026-07 개편**: 구 검색 URL(`List_GI/GI_Search_Keyword.asp`)이 키워드를 무시하고
  전체 공고판 최신 40건으로 리다이렉트 → 재무 공고가 우연히 40건 안에 있는 날만
  수집되는 복불복(격일 0건↔수집 플랩, 마감 오판 반복의 근본 원인).
- 실제 검색은 XHR: `POST https://www.gamejob.co.kr/Recruit/_GI_Job_List/`
  (form: `isDefault=true, condition[searchtype]=all, condition[searchstring]=키워드(UTF-8),
  condition[menucode]=, condition[tabcode]=1, page, direct=0, order=1, pagesize=40, tabcode=1`,
  헤더 `X-Requested-With: XMLHttpRequest`, 토큰·쿠키 불요).
- 응답은 `div.jobListWrap` HTML 조각. **컨테이너가 안 잡히면 0건이 아니라 수집 실패로
  승격**시켜 마크업 재개편을 즉시 가시화한다.
- 상세 `<title>`의 첫 대괄호는 회사명이 아니라 "[전략실]"·"[Finance Div.]" 같은
  부서명인 공고가 많다 — **회사명·제목은 목록 행이 1차 소스**.

### 잡코리아
- 검색(`/Search/?stext=`)과 기업페이지(`/company/{id}/recruit`)는 정적 HTML로 정상.
- **GI 상세(`/Recruit/GI_Read/{gno}`)는 React SPA로 전환**(2026-07-09 확인) — 정적
  HTML에도, 렌더링된 DOM에도 JD 텍스트가 없다. 그러나 **상세요강은 별도 iframe
  (`/Recruit/GI_Read_Comt_Ifrm?Gno={gno}`)에 서버 렌더링(SSR)으로 존재**하며 일반
  requests로 수집된다(2026-07-16 라이브 누적 13/13건 성공: 스마일게이트·넷마블·웹젠·
  펄어비스·시프트업·잡코리아 검색 수집분, 텍스트 415~1,359자). 과거 "이미지 JD라 OCR 필수" 결론은
  이 iframe을 못 본 것. 수집은 `src/scraper/jobkorea_detail.enrich_gi_postings`
  (파이프라인 후처리, 신규 요청 cap 12건/런 + 지터)가 담당하고, 어댑터의
  `fetch_detail=False`(목록은 제목 기반)는 그대로 유지된다.
- DB에 본문이 이미 확보된 GI 공고는 재요청하지 않고 저장본을 실어 준다 — 오늘
  수집분이 제목뿐이어도 upsert가 본문 변경(MODIFIED)으로 오판하지 않게 하기 위함.
- 기업페이지 company_id는 폐지·재배정될 수 있어 **회사명 교차검증**(`_verify_company`)
  실패 시 전량 폐기한다(과거 실사고: '빅게임스튜디오' id가 바이오회사로 재배정).
- 기업페이지는 과거 마감 공고도 나열한다 — `마감 (~날짜)` 배지는 수집 제외(좀비 방지).
- IP 차단은 러너(머신) 단위로 온다. 같은 프로세스 안의 재시도로는 복구되지 않으므로
  CI는 실패 마커(`data/last_failed_sources.txt`) 감지 시 **새 러너에서 2차 시도**한다.

### greetinghr (카카오게임즈·111퍼센트·슈퍼센트·에피드게임즈)
- 목록 API: `GET https://api.greetinghr.com/ats/v1.1/career/workspaces/{wid}/openings`
  — `dueDate`(예: `2026-08-02T14:59:59Z` = KST 자정 직전)가 마감일. 단 `2033-…`처럼
  1년 이상 남은 값은 사실상 상시채용이라 None 처리.
- 상세 API(`/openings/{id}`)는 404. 대신 **공고 페이지가 SSR**
  (`https://{sub}.career.greetinghr.com/ko/o/{id}`) — `[class*='Content']` 중 최대
  텍스트 블록이 JD 본문(클래스 해시는 빌드마다 바뀌므로 부분 일치로 잡는다).

### 사람인
- 목록 `.job_date` 배지("~ 07/31(금)"·"D-N"·"오늘마감"·"상시채용")를
  `src/utils/dateparse.parse_deadline_badge`로 환산(게임잡과 공용).
- **상세는 반드시 `jobs/view?rec_idx=`(정규 주소)로 간다.** `jobs/relay/view?rec_idx=`는
  상세요강이 없는 중계 페이지다 — 셀렉터가 전부 빗나가고 `body` 폴백이 걸려
  **전 공고가 똑같은 4,999자 사이트 네비게이션**을 본문으로 저장했다(2026-07-21 발견:
  사람인 41행 전부 raw_html SHA-256 동일. 그 결과 대리·과장급 공고가 '신입 0~1년',
  제목에 '[경력]'이 있는데 '경력 무관'으로 저장돼 있었다).
- 본문 셀렉터는 `.wrap_jv_cont` → `.jv_detail` → `.user_content` 순.
  **`body` 폴백을 다시 넣지 말 것** — 공통 레이아웃이 본문으로 둔갑하면 길이가 길어
  `has_body`가 참이 되고, 대시보드가 '미확인'으로 감추지도 못해 근거 없는 분류값이
  사실처럼 노출된다(길이가 긴 쓰레기 > 짧은 진실).
- `.wrap_jv_cont`에는 상세요강뿐 아니라 **요약표(경력·학력·근무형태·급여)**가 함께 들어
  있어 연차 추출의 1차 근거가 된다. 라이브 실측 12/12건 확보(5,961~7,854자, 전부 고유).
- 과거 오염 행 정리용 일회성 스크립트: `scripts/clean_saramin_boilerplate.py`
  (dry-run 기본, `--apply` 시 `.bak` 백업 생성).

## 파이프라인 구조 결정

- **중복의 2계층 처리**: ① 같은 잡코리아 GI를 검색·기업페이지가 이중 수집한 것은
  파이프라인에서 병합(`main.dedupe_jobkorea_gi`, DB에 1벌만). ② 소스가 아예 다른
  같은 공고(공식 vs 플랫폼)는 **DB에 양쪽 다 보존**하고 표시 계층(텔레그램·대시보드)
  에서 병합한다 — "누락 0 우선" 원칙 + 멀티소스 링크 기능 보존. 병합 키는
  `content_key`: 정규화 회사명+제목, 제목 앞 `[회사명류]` 프리픽스가 회사명과
  포함관계면(컴투스 ⊂ 컴투스홀딩스) 회사키로 승격. 대표 카드는 공식 소스
  (`source_rank`: 플랫폼 4종 외 전부 공식) 우선.
  키 정규화 2종 추가(2026-07-21): ① **회사 별칭**(`_COMPANY_ALIASES`) — 수집처마다
  영문 사명/한글 법인명이 갈리는 실측 쌍만 보수적으로 등재(`nx3games` → `엔엑스쓰리게임즈`).
  ② **남은 대괄호는 벗기기만** — 괄호쌍만 없애고 안의 낱말은 남긴다. 잡코리아
  '[NX3GAMES] 전략실 회계담당자 (주니어)'와 게임잡 '[전략실] 회계담당자 (주니어)'가
  한쪽만 회사 프리픽스가 벗겨져 키가 어긋나던 것을 해소. 낱말이 남으므로
  '[신입] …'과 '[경력] …'은 여전히 별개 키다.
  ⚠️ 대표 카드 교체는 아직 **레코드 전체를 공식 소스로 갈아끼운다**(dashboard JS의
  `Object.assign`) — 공식 어댑터가 본문을 못 가져온 공고(컴투스·위메이드 등 20자대)는
  본문 있는 게임잡 분류값을 버리게 된다. 필드별 품질 기준 병합은 개선 대기(P1).
- **마감(CLOSED) 판정의 4중 방어**(delta_analyzer): ① 오늘 전체 수집 3건 미만이면
  전면 보류 ② 수집 실패 소스의 공고 보존 ③ '성공했지만 0건'인 플랫폼(suspect)의
  공고 보류 — 4개 재무 키워드 검색이 전부 0건인 것은 공고 전멸보다 검색 오동작일
  개연성이 압도적(게임잡 플랩 실측) ④ 일괄 소멸 가드 — 기업 어댑터의 활성 공고
  2건 이상이 한 번에 전부 미수집되거나 4개 소스 이상이 같은 날 동시 전멸하면
  사이트 개편 의심으로 보류(`last_mass_close_held`로 노출 → 텔레그램 경고).
  활성 1건짜리 소스의 일상적 마감은 그대로 진행되므로 좀비화되지 않는다.
- **수집량 급감 감지**: scrape_logs.source_counts(JSON)에 소스별 수집 건수를 매 실행
  저장하고, 플랫폼이 기준선 평균 대비 30% 미만(0건 제외 — 0건은 별도 경고)이면
  📉 경고(main.py 8-4). 기준선은 **오늘을 제외한 달력 7일**([오늘-7, 오늘))이며
  같은 날 재실행은 소스별 최댓값 하나로 축약된다(get_recent_source_counts) —
  '최근 7회 실행' 기준이던 시절 같은 날 검증 반복이 기준선을 독점하던 결함의 교정
  (2026-07-24, 5598d58). 관측 3일 미만이면 비교를 쉬고 INFO 로그만 남긴다.
- **경고는 '하루 기준', 가드는 '시도 기준'**: 최종 브리핑은 재시도 실행이 보내는데,
  1차가 정상 수집한 소스를 재시도 러너만 차단당해 '접속 실패 24곳' 오경보가 나간
  실사고(2026-07-16, run 84) 교정 — scrape_logs.successful_sources(JSON)로 오늘
  성공 이력을 조회해(`get_sources_succeeded_today`) 접속 실패·0건 경고에서 제외한다
  (main.py 13-a). 단 마감 보류 등 데이터 보호 가드는 시도별 보수 판정을 유지한다.
- **마감 판정의 5번째 방어 — 연속 미관측 유예**(delta_analyzer.CLOSE_GRACE_DAYS=2,
  2026-07-24): 검색 결과가 실행마다 출렁여 공고가 하루 빠졌다 돌아오는 플랩(사람인
  run 77 실측: 마감 3건이 1분 뒤 부활)을 막는다. 오늘 관측된 공고는
  job_postings.last_seen_date에 스탬프되고, 마지막 관측일로부터 2일이 지나야 CLOSED
  확정. 같은 날 재실행 누락은 어떤 경우에도 마감되지 않는다. **대가: 진짜 마감의
  감지·텔레그램 알림이 최대 1일 늦는다**(의도된 트레이드오프).
- **외부 watchdog(dead man's switch)** — 워크플로 자체가 안 뜨는 완전 무음을 밖에서
  감시(2026-07-24). 브리핑 발송 성공(또는 주말·공휴일 의도적 스킵) 시
  HEALTHCHECK_PING_URL로 ping을 보내고, 신호가 끊기면 감시 서비스가 이메일 경보.
  시크릿 미설정이면 기능 자체가 조용히 꺼져 있다. 설정: docs/WATCHDOG_SETUP.md.
- **'알려진 차단'은 경고가 아니라 정보로 내린다**(src/utils/known_blocks.py, 2026-07-22).
  우리 코드로 고칠 수 없는 실패(현재: 원티드 = 러너 IP가 원티드 WAF에 차단)가 매일
  경고로 뜨면 '경고 없으면 할 일 없음' 원칙 자체가 무뎌진다. 그렇다고 숨기면 차단이
  풀린 것도 모른다. 그래서 3단으로 다룬다 — 평소 `ℹ️ 조치 불요` 정보 줄 / 다시 수집되면
  `✅ 차단 해제 확인`(등록 해제 필요) / 마지막 성공일로부터 `ZOMBIE_ALERT_DAYS`(14일)
  경과하면 `⚠️ 수동 확인 필요` 경고로 승격.
  **부수 효과(의도한 것)**: 알려진 차단은 재시도 게이트(`data/last_failed_sources.txt`)에서
  제외되므로 2차 러너 재실행이 예약되지 않는다. 원티드가 상시 실패하는 바람에 매일
  파이프라인이 통째로 두 번 돌고 있었고(당일 통계 이중 합산의 상시 원인), 이것이 사라진다.
  **고칠 때 주의**: 텔레그램 발송 보류 게이트(`SUPPRESS_ALERT_ON_SOURCE_FAILURE`)는
  반드시 `retry_targets` 기준이어야 한다. `failed_sources` 기준으로 두면 1차가 보류하는데
  2차는 예약되지 않아 **브리핑이 통째로 사라진다**.
  **차단 여부 재확인**: 수동 실행 워크플로 `Debug - Wanted Access Probe`
  (scripts/probe_wanted_api.py)를 dispatch하면 러너에서 3경로를 찔러 결과만 출력한다
  (DB·index.html 미변경).
- **posted_at = 수집일**이다(원 공고 등록일 아님). 텔레그램 '신규' 판정이
  `posted_at == 실행일`에 의존하므로 이 의미를 바꾸면 신규 알림이 깨진다.
- **deadline은 None이면 기존 값 보존**(upsert의 COALESCE) — 마감일을 안 주는 소스가
  이미 확보된 마감일을 지우지 않도록.
- **본문 축소 방지 가드**(upsert): 오늘 수집분이 기확보 본문 대비 '열화'면
  (`src/utils/jdtext.body_degraded` — 마커 소실+70% 미만, 또는 400자 이상 본문의
  1/3 이하 붕괴) 기존 본문을 보존하고 MODIFIED로 잡지 않는다. 분량 70% 이상이
  유지되는 개정은 마커가 사라져도 반영(영문 전환 등 구본 고착 방지). 가드 발동 시
  **posting dict에도 보존본을 되돌려** 분류기(job_categories)까지 보호한다.
  마커(`JD_MARKERS`)는 실본문 전용 4종 — '지원자격'·'모집요강'은 잡코리아 SPA
  껍데기 라벨에도 있어 제외(실측: 검색 수집분 4/4 오검출 이력). 보강(enrich)과
  보존(guard)이 같은 기준을 쓰도록 jdtext.py로 단일화했다.
- **마감일 변경 상세**: upsert가 '기존 마감일이 실제로 바뀐' 경우만
  `last_change_details`에 전후를 남기고(최초 확보는 제외), 텔레그램 '🔄 마감일 변경'
  섹션이 연장/단축을 표시한다.
- **재공고(🔁) 판별**(`src/utils/dedup.compute_repost_flags`): 현 활성 그룹의 최초
  관측이 같은 content_key의 마지막 CLOSED 관측보다 **2일 이상** 뒤일 때만 재공고 —
  한쪽 소스만 닫혔던 좀비 CLOSED(플랩 이력)는 그룹 최초 관측이 앞서므로, 마감
  직후 하루 이틀 내 재등장(gno 갱신·플랫폼 이동·플랩)은 공백 규칙으로 각각 오탐을
  막는다. 같은 실행 안의 구공고 마감+신공고 등장은 보수적으로 미표시(의도).
  플래그는 Python에서 계산해 데이터로 주입하므로 대시보드 JS 미러링이 필요 없다.
- **DB 마이그레이션**은 `_add_column_if_not_exists`로 점진 추가만 한다(파괴적 변경 금지).
- **시각은 전부 KST**(`src/utils/timeutil`) — CI 러너가 UTC라 naive now()를 쓰면
  신규/변경 판정이 하루 밀린다(실사고 이력).

## 분류(hybrid_engine) 규칙 요약

- 재무 판별은 **제목만** 사용한다 — 본문 매칭은 'ir'∈hiring, '감사합니다' 오탐이
  심해 폐기(라이브 53/53 전부 오탐이었던 이력).
- 개발직 차단: 제목에 개발자·개발 담당·시스템 개발·엔지니어 등이 있으면 재무 부서
  소속이어도 제외. 단 "내부회계 IT 담당자"류(회계 유관 IT)는 의도적으로 통과.
- **연차 추출은 탐색 '범위'를 좁은 순서로 훑는다**(`extract_experience`):
  제목 → 수집처 '경력' 요약 라벨 → 자격요건 구역 → 전체 본문 → 직급 폴백.
  본문 전체를 한 번에 훑으면 **본문 뒤쪽에 붙는 '이 기업의 다른 공고' 목록**의 연차를
  요건으로 오독한다(2026-07-21 실측: 게임잡 컴투스 본문 8,600자 이후 타 공고의
  '4-8년차'를 읽어, IR/공시 **주니어와 시니어가 똑같이 4~8년**으로 저장. 제목이
  '10년 이상'인 팀장 공고도 4~8년이었다).
- '경력' 요약 라벨(`_EXP_LABEL_RE`)은 게임잡("경력   경력 2년 이상")·사람인
  ("경력\\n경력 5~12년")이 상단 요약표에서 **라벨과 값에 '경력'을 겹쳐 쓰는** 성질을 이용한다.
  본문 하단 타 공고 목록은 값만("경력 4년↑") 있어 이 이중 표기가 없으므로 첫 매치가
  항상 해당 공고 자신의 요건이 된다.
- 제목의 '신입'(단독) 신호는 여전히 본문보다 우선 — 본문의 "경력 개발 기회" 같은
  문구가 신입 판정을 무력화하지 않도록.
- 지원 표기: `3년~5년`·`1~3년차`·**`7-10년`(앞 '년' 생략)**·`3년 이상`·`5년 이하`·
  **`5년 ↓`(사람인 요약표)**·`3년 전후`·`경력 3년`. 범위는 `_exp_range`가 상한 40년으로
  거른다(`2024-2025년` 같은 연도 표기가 연차로 둔갑하는 것 방지).
- 라벨 자리의 '무관'(사람인 '무관(신입포함)')만 경력 무관으로 인정한다 — 본문 전체에서
  '무관'을 인정하면 '학력 무관'을 경력으로 오독한다.
- ⚠️ **근무형태는 아직 '기본값 전면출근'이다**(재택 키워드가 없으면 전면출근 반환).
  즉 화면의 '전면출근'은 근거가 아니라 미판정일 수 있다 — 개선 대기(P1).
